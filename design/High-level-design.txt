Great questions — let me clarify each field by grounding it directly in the **CycloneDX 1.4 spec**.

---

### 1. `SpecVersion`
The version of the **CycloneDX format itself** — not your tool's version or the scanned project's version. It tells parsers and validators which schema to validate against.

For this assignment: always the hardcoded string `"1.4"`.

---

### 2. `Version`
The **serial version of this particular BOM document**. It starts at `1` and is meant to increment each time the BOM is regenerated for the same project (to support diffing BOMs over time).

For this assignment: always hardcoded to `1`. No tracking needed.

---

### 3. `Metadata`
Describes **the context in which the BOM was generated** — not the dependencies, but the tool and the subject project.

```go
type Metadata struct {
    Timestamp string    // ISO-8601, e.g. "2024-03-16T10:00:00Z"
    Tools     []Tool    // what generated this BOM
    Component Component // the root project being scanned (optional but good practice)
}

type Tool struct {
    Vendor  string // "cpp-sbom-builder"
    Name    string // "cpp-sbom-builder"
    Version string // your tool's version, e.g. "1.0.0"
}
```

---

### 4. `Components`
The **list of detected third-party dependencies** — this is the core payload of the BOM. Each entry maps from your internal `Dependency` struct.

```go
type Component struct {
    Type       string // always "library" for dependencies
    Name       string // e.g. "openssl"
    Version    string // e.g. "1.1.1" or "unknown"
    PackageURL string `json:"purl"` // e.g. "pkg:generic/openssl@1.1.1"
    BOMRef     string // unique ID within this doc, e.g. "openssl-1.1.1"
    Evidence   ComponentEvidence // optional but valuable
}

type ComponentEvidence struct {
    Occurrences []Occurrence // where in the project it was found
}

type Occurrence struct {
    Location string // file path that triggered detection
}
```

---

### How it fits together

```json
{
  "bomFormat": "CycloneDX",
  "specVersion": "1.4",
  "version": 1,
  "metadata": {
    "timestamp": "2024-03-16T10:00:00Z",
    "tools": [{ "vendor": "cpp-sbom-builder", "name": "cpp-sbom-builder", "version": "1.0.0" }],
    "component": { "type": "application", "name": "my-cpp-project", "version": "unknown" }
  },
  "components": [
    {
      "type": "library",
      "name": "openssl",
      "version": "1.1.1",
      "purl": "pkg:generic/openssl@1.1.1",
      "bom-ref": "openssl-1.1.1"
    }
  ]
}
```

The `Evidence.Occurrences` field is optional per spec but I'd strongly recommend including it — it directly addresses the **false positive guiding question** by showing reviewers *why* each component was included.

High-Level ArchitectureThe project follows a Strategy Pattern managed by a central Orchestrator.CLI Layer: Handles flags, logging, and output piping.Engine (Orchestrator): Manages the file walking, dispatches work to strategies, and aggregates results.Strategies: Modular components that know how to "identify" a dependency from a specific source (CMake, Headers, etc.).SBOM Formatter: Converts the internal dependency model into a valid CycloneDX/SPDX JSON.2. Key Components & ResponsibilitiesComponentResponsibilityEngineWalks the filesystem once. For each file, it queries registered strategies.Strategy (Interface)CanHandle(path) and Extract(path). Decouples parsing logic from file walking.ResultAggregatorCollects findings from concurrent workers and performs de-duplication.ScannerA worker-pool implementation that limits OS resource usage.3. Data FlowInitialize: The Engine registers a list of Strategies (e.g., CMakeStrategy, IncludeStrategy).Walk: filepath.WalkDir starts. For every directory/file, a job is sent to the Worker Pool.Detect: A worker runs all Strategies against the file. If a strategy matches, it returns a Dependency object.Collect: Results are sent via a channel to the Aggregator.Deduplicate: The Aggregator uses a map (Key: Name + Version + Path) to ensure uniqueness.Finalize: The map is converted to the final SBOM JSON.4. Core Data StructuresThe Strategy InterfaceGotype Strategy interface {
    Name() string
    // Filter: Is this file relevant to this strategy?
    Match(path string, isDir bool) bool
    // Logic: Parse the file and return found components
    Extract(ctx context.Context, path string) ([]Component, error)
}
The Component ModelGotype Component struct {
    Name        string
    Version     string // "unknown" if not found
    Type        string // e.g., "library", "framework"
    FoundBy     string // Which strategy found it?
    Location    string // Path to the file/folder
    Confidence  float64 // (Optional) Senior touch: 1.0 for manifest, 0.5 for header
}
5. Concurrency ConsiderationsIn Go, the "Senior" way to handle 10GB of files is to use a Fixed Worker Pool or a Semaphore to avoid hitting ulimit (max open files).The Pipeline: * One Goroutine for WalkDir (Producer).N Goroutines (Workers) for Extract.One Goroutine for Aggregation (Consumer).Synchronization: Use a sync.WaitGroup to know when workers are done and a Context for graceful cancellation (e.g., if the user hits Ctrl+C).6. Performance ConsiderationsEarly Exit in Match: Strategies should check file extensions (e.g., .cmake) before opening the file.Buffered Reading: Use bufio.NewScanner to read headers. Don't use os.ReadFile for large .cpp files; stop reading after the first 100 lines if looking for #include.Path Pruning: Don't walk into .git/, node_modules/, or build/ directories.7. Simplifications for Take-HomeRegex over AST: Don't write a full C++ parser. Use optimized Regex for #include <...> and find_package(...).In-Memory Aggregation: For a single run, a map[string]Component in memory is perfectly fine; no need for a database.Minimalist CycloneDX: Implement only the "Required" fields of the CycloneDX spec (BomFormat, SpecVersion, Components) to save time.Local Sample Data: Instead of downloading a real 10GB repo, provide a script that generates a "Mock" 10GB structure with empty files to prove performance.
